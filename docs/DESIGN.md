# 共進化の砦（仮）— 設計ドキュメント v3

> **LAN対応の2人協力プレイ・タワーディフェンス。敵は小さなニューラルネットを持ち、世代を経るごとにプレイヤーのタワー配置を学習して経路を最適化してくる。プレイヤー2人は別々のPCから接続して「建築」と「前線」に役割分担し、敵の進化を出し抜く戦略を練る。さらに追加機能として、互いに敵を送り込み合う対戦モードを搭載。**

**v3の主な変更点**：ディレクトリ構造をフラットからサブパッケージ分離型に再編。担当領域を物理的に分離してマージ衝突を防ぐ設計に。

---

## 1. ゲームコンセプト

### 1.1 ゲーム概要

* ジャンル：タワーディフェンス × 進化計算 × LANネットワーク協力プレイ
* プレイ人数：2人（**別々のPC、同じLAN内で接続**）
* プレイモード：
    * **協力モード（メイン）**：2人で1つの砦を守る
    * **対戦モード（追加機能）**：2人が別々の砦を持ち、敵を送り込み合う
    * **soloモード（フォールバック）**：1台で2プレイヤー操作（デバッグ用）
* ゲームのコア体験：「敵が学習してくる」緊張感 × 2人の役割分担 × ネット越しの連携

### 1.2 「すごい」ポイント（他班との差別化）

1. **敵が遺伝的ニューラルネットで進化する**：単純な強さ調整ではなく、敵の「動き方」そのものが世代ごとに最適化される
2. **LAN協力プレイ**：他班でやる班はほぼいない、技術アピール強
3. **対戦モードあり**：進化AIと相性が良い（互いの戦法を学ぶ敵が育つ）
4. **進化の可視化**：世代ごとの敵の動きの違いをグラフ・系統樹で見せる演出

### 1.3 勝敗条件

* **協力モード**：規定ウェーブ数（例：20ウェーブ）を生き残る／拠点HP0で敗北
* **対戦モード**：相手の拠点HPを先に0にしたら勝利

## 2. ネットワーク設計

### 2.1 アーキテクチャ：クライアント-サーバ型（権威サーバ）

```
┌─────────────────────┐         ┌─────────────────────┐
│  Host (Player 1)    │         │  Client (Player 2)  │
│  ─────────────────  │  UDP    │  ─────────────────  │
│  ・ゲーム状態を保持 │ ←─────→ │  ・ゲーム状態を受信 │
│  ・進化AI計算       │         │  ・操作を送信       │
│  ・物理・当たり判定 │         │  ・描画のみ実行     │
│  ・正本データを持つ │         │                     │
└─────────────────────┘         └─────────────────────┘
       権威サーバ                       純クライアント
```

**重要な原則：**
* **ゲームロジックは全部ホスト側で実行**（"authoritative server" モデル）
* クライアントは「自分の操作を送信」と「受信した状態を描画」しかしない
* これにより**状態のズレ（desync）が原理的に発生しない**
* 進化AIの計算もホスト側のみ → 結果一意性が保証される

### 2.2 通信プロトコル：UDP

#### なぜUDPか
* TDは1秒間に複数回、敵の位置情報を送る必要がある
* 「最新の位置」が届けばよく、**古い位置データは捨ててOK**
* TCPだと再送のせいで遅延が蓄積する（特にWi-Fi環境）
* UDPなら届かなかったパケットはスキップ、次の最新が来る

#### UDPの弱点と対策
| 弱点 | 対策 |
|------|------|
| パケットが届かない | 重要操作（タワー設置等）はシーケンス番号付きで再送 |
| 順序が逆転する | シーケンス番号で古いパケットを破棄 |
| 接続概念がない | 独自のハンドシェイク・タイムアウト機構を実装 |

### 2.3 メッセージ形式

すべてのメッセージはJSON形式（`json.dumps()` → bytes化）でやり取り。

#### クライアント → ホスト（操作送信）
```json
{
  "type": "input",
  "seq": 142,
  "player_id": 2,
  "input": {
    "move": [0.5, -1.0],
    "attack": true,
    "skill": false,
    "place_tower": null
  }
}
```

#### ホスト → クライアント（状態同期）
```json
{
  "type": "state",
  "seq": 1024,
  "tick": 18420,
  "enemies": [
    {"id": 17, "x": 320, "y": 580, "hp": 80, "type": "basic"}
  ],
  "towers": [
    {"id": 3, "x": 200, "y": 400, "type": "fire", "level": 2}
  ],
  "players": [
    {"id": 1, "x": 100, "y": 200, "hp": 100},
    {"id": 2, "x": 800, "y": 400, "hp": 100}
  ],
  "core_hp": 950,
  "resource": 250,
  "wave": 5,
  "generation": 4
}
```

#### 接続確立（ハンドシェイク）
```json
// client → host
{"type": "connect", "name": "Player2"}

// host → client
{"type": "connect_ok", "player_id": 2, "config": {}}
```

#### イベント通知（重要・確実に届けたい）
```json
{
  "type": "event",
  "seq": 305,
  "event": "tower_placed",
  "data": {"x": 300, "y": 400, "type": "fire", "tower_id": 7},
  "ack_required": true
}
```
※ `ack_required: true` のメッセージは、受信側がACKを返さない場合に再送

### 2.4 送信頻度の設計

| メッセージ種別 | 頻度 | 信頼性 |
|----------------|------|--------|
| 状態同期（state） | 20Hz（50msに1回） | 不要（最新が届けばOK） |
| 入力送信（input） | 30Hz（33msに1回） | 不要 |
| イベント（event） | 発生時 | **ACK＋再送で確実に** |
| ハートビート（ping） | 2Hz | タイムアウト検知用 |

### 2.5 クライアント側の補間（interpolation）

20Hzの状態同期だと敵がカクカク動くので、**クライアント側で線形補間**して描画する：

```
受信時刻 t1 における敵位置: (320, 580)
受信時刻 t2 における敵位置: (340, 590)
描画時刻 t (t1 < t < t2) における敵位置:
    線形補間で (320 + (340-320) * (t-t1)/(t2-t1), ...)
```

これにより60FPSで滑らかに見える。

### 2.6 接続の流れ

```
1. Host:   "python main.py --host" でサーバ起動、IPを表示
2. Client: "python main.py --client --ip=192.168.1.10" で接続要求送信
3. Host:   接続要求を受け、player_id=2 を割り当てて connect_ok を返信
4. Client: connect_ok 受信、ゲーム画面に遷移
5. 両者:   状態同期・入力送信のループに入る
6. 切断:   一定時間ハートビート途絶 → タイムアウト処理
```

### 2.7 soloモード（フォールバック）

* `python main.py --solo` で 1人プレイ可能
* この場合、ホスト側で両プレイヤーをローカル操作（マウス＋キーボードで両方動かす）
* **デバッグ・テストで重要**：ネット越しじゃないとプレイできないと開発効率が落ちる
* 発表当日にネットワークが不安定だった場合のフォールバックでもある

## 3. ゲームシステム詳細

### 3.1 マップ・配置

* マップ：固定サイズ（1280×720px）
* 敵の出現口：マップ端の複数箇所
* 拠点（コア／fortress）：マップ中央 or 下端
* タワー：**マップ上どこにでも自由配置可能**
* 通行不可エリア：タワー本体の周辺のみ

### 3.2 敵

* 各敵が小さなNNを持って動きを決定
* HPなどのステータスは固定／敵タイプで差別化
* **進化するのは「動き方（経路選択）」のみ**

### 3.3 ウェーブ進行（協力モード）

1. 準備フェーズ（数十秒）
2. 戦闘フェーズ（敵スポーン〜全滅 or 拠点到達）
3. 集計フェーズ（適応度評価）
4. 進化フェーズ（次世代NN生成）
5. → 1へ

### 3.4 対戦モード（追加機能）

* 画面分割（1人1台のPCに自分の砦を表示）
* 敵を**相手の砦に送る**ボタン／コマンド
* 送る敵は自分のNNプールから選択
* **「相手の戦法を学習した敵を送り合う」**心理戦が生まれる
* 勝敗：先に相手の拠点HPを0にした方が勝利

### 3.5 経済システム

* **共有リソース**（協力モード）／個別リソース（対戦モード）
* 入手：敵撃破、ウェーブクリアボーナス
* 使用：タワー設置・アップグレード・修理・スキル

## 4. 進化AIの設計

### 4.1 ニューラルネット構造（敵1体ごと）

* **入力層**（12ノード）
    * 自分のHP（正規化）
    * 拠点までの距離・方向（dx, dy）
    * 近傍タワー3つの相対位置と種類（3×3=9）
* **隠れ層**：1〜2層、各8〜16ノード（tanh活性化）
* **出力層**（2ノード）：vx, vy（-1〜+1）

### 4.2 遺伝的アルゴリズム

#### 適応度
```
fitness = (拠点に与えたダメージ) × 10
       + (生存時間) × 1
       + (拠点までの最短到達距離の改善) × 5
```

#### 選択・交叉・突然変異
* エリート選択：上位20%はそのまま次世代へ
* トーナメント選択：残りは3〜5体から最良を選ぶ
* 一様交叉：親2体の重みを要素ごとに50%確率で混合
* 突然変異：確率5%、ガウスノイズを重みに加算

### 4.3 実装ライブラリ

* **NumPyのみ**で実装（PyTorch/TensorFlow不要）
* 学習はリアルタイムではなく**ウェーブ間バッチ処理**
* **ネットワーク経由ではNN自体は送らない**：ホスト側で計算した結果（敵の位置）のみクライアントに送る

### 4.4 ネットワークと進化AIの関係

```
[ホスト側]
  ウェーブ終了
  → 適応度計算
  → 次世代NN生成（NumPy計算）
  → 新ウェーブ開始
  → 毎フレーム：NN推論で敵位置更新
  → 20Hzで「敵の位置だけ」をクライアントに送信

[クライアント側]
  敵の位置を受信して描画するだけ
  NN自体は知らなくていい
```

→ NN自体は通信に乗らないので、**通信負荷が極めて軽い**

### 4.5 デモ用機能

* ウェーブ1の敵と最新ウェーブの敵を並べて再生
* 適応度の世代別グラフ表示
* （余裕があれば）進化系統樹の可視化

6. **`assets/` で素材一元管理**

### 6.2 ネットワーク層のクラス詳細

```
NetServer
├─ socket: UDPソケット
├─ clients: {addr: PlayerInfo} 接続中クライアント
├─ recv_thread: 受信スレッド
├─ send_state(): 全クライアントに状態をブロードキャスト
├─ handle_input(addr, msg): クライアント操作を処理
├─ handle_event(addr, msg): イベント受信処理（ACK返信）
└─ heartbeat_check(): タイムアウト検知

NetClient
├─ socket: UDPソケット
├─ server_addr: ホストのアドレス
├─ seq_counter: 送信シーケンス番号
├─ recv_thread: 受信スレッド
├─ pending_events: ACK待ちイベント（再送用）
├─ send_input(input_dict): 操作送信
├─ send_event(event_dict): イベント送信（ACK必須）
├─ on_state_received(state): 状態バッファに格納
└─ heartbeat_send(): 定期ping
```

### 6.3 設計方針

* **継承よりコンポジション**を優先（タワー・敵の派生は例外）
* インスタンス変数は `self._変数名`、外部アクセスは getter/setter経由
* `pygame.sprite.Sprite` 継承は描画対象クラスのみ
* **スレッド安全性**：ネットワーク受信スレッドとメインスレッドの間は **キュー（`queue.Queue`）で受け渡し**

## 8. コード規約（全員遵守、詳細は AGENTS.md 参照）

### 8.1 命名規則

* クラス名：`PascalCase`
* 関数・メソッド名：`snake_case`

import os
